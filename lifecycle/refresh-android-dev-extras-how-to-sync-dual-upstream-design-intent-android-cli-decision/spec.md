# Specification: Refresh android-dev-extras HOW-TO-SYNC (dual upstream, design intent, android-cli vendoring + guard)

> Epic context: [`research/android-dev-extras-registry/research.md`](../../research/android-dev-extras-registry/research.md) — this ticket executes the epic's DR-1 option F (vendor with detect-then-load guard), DR-2 (dual-upstream documentation), and DR-3 (design intent captured at plugin level) as a single coherent PR.

## Problem Statement

`plugins/android-dev-extras/HOW-TO-SYNC.md` has two documented gaps identified by the discovery round: (a) it is silent on plugin-level motivation, so new curators encounter no record of why this plugin exists or what its curation bar is; (b) it references only `github.com/android/skills` as upstream, despite the authoritative source being `dl.google.com/dac/dac_skills.zip` (the DAC bundle; the GitHub repo is a public overlay/mirror). Both gaps manifested concretely: the `android-cli` skill — dac-only, missing from the public GitHub browse curation — was never considered for vendoring. This spec refreshes HOW-TO-SYNC to close both gaps and executes the android-cli vendoring with a Claude-specific detect-then-load guard, documenting the guard as the first accepted divergence from the verbatim-mirror policy. Future refreshes catch dac-only skills via a new `android skills list` reconciliation check-step; curators encounter plugin motivation in the doc they already read during sync; Claude Code users gain a per-project toggle for the Android CLI skill.

## Requirements

All 12 requirements are Must-have: they form a single coherent atomic PR (dual-upstream documentation, design-intent capture, the android-cli vendoring with guard, and the sync-procedure divergence-preservation branch) — omitting any one leaves the plugin in a partially-documented or partially-patched state that is worse than the status quo. The guard mechanism (Req 6) is known to be advisory (relies on model compliance, not loader-level gating); the spec accepts that limitation explicitly and compensates with the divergence-preservation requirement (Req 12) and the workflow cross-references (Req 8, Req 9) — no single mechanism is load-bearing.

## Non-Requirements

- No drift-detection script, scheduled CI job, or automated sync tool. The `android skills list` reconciliation in Requirement 3 is a manual step invoked during sync.
- No spike, investigation, or upstream bug-filing on DAC URL stability. The spec records "undocumented URL; fallback to GitHub" as the policy; no attempt is made to obtain a stability guarantee.
- No external-consumer policy, breaking-change policy, or deprecation mechanism for the plugin. The marketplace repo is public but this ticket treats it as author-owned curation surface.
- No trigger-phrase tuning, CFA-vocabulary rewriting, or description rewording for the vendored `android-cli` SKILL.md beyond the guard block and inline marker.
- No `/lifecycle` integration, hook additions, or slash-command declarations in the vendored skill.
- No plugin-level `README.md` for `android-dev-extras`. Design intent lives in HOW-TO-SYNC.md (Requirement 1).
- No changes to the other three marketplace plugins (`cortex-dev-extras`, `cortex-pr-review`, `cortex-ui-extras`).
- No changes to `chickfila-android`'s `.claude/settings.local.json` permission list. Write-style `android` subcommands (`screen capture`, `emulator`, `run`) will prompt on first use in that downstream — acceptable.
- No changes to `.claude-plugin/plugin.json` for `android-dev-extras` beyond what already exists.
- **No divergence-budget policy codification** beyond a single revisit threshold (see below). This ticket accepts the first divergence (android-cli guard) without committing to a formal policy for when the list should stop growing. Revisit threshold: if `### Accepted divergences` accumulates 3+ entries, OR if any single vendored skill accumulates 2+ distinct patches, reopen the vendor-vs-skip decision from the epic research. Below that threshold, per-entry evaluation is acceptable. This restores the DR-4 hold posture (which the research noted collapsed when DR-1 chose option F instead of option C) with a concrete re-evaluation trigger.
- **No runtime version-compatibility mitigation**. The detect-then-load guard (Req 6) checks `command -v android` — presence of the binary on PATH — not version compatibility with the vendored `android help` dump (pinned to CLI `0.7.15232955` per the upstream v0.0.2 snapshot). A user running a materially different CLI version passes the guard and sees vendored content that may reference subcommands/flags that no longer match. This is acknowledged in Edge Cases but not remediated; remediation would require either a version-aware guard (not currently supported by Claude Code's documented mechanisms) or an out-of-band CI check (explicitly excluded above).

## Edge Cases

- **`android` CLI not installed**: the `!`command -v android`` preamble injects `NOT_INSTALLED` as its output; the adjacent prose instruction tells Claude to abort all `android`/`adb` flows in the skill. Expected behavior (advisory, not loader-enforced — see Req 6): Claude reads the marker, honors the abort instruction, and does not invoke a missing binary. If Claude ignores the abort instruction, the downstream `Bash` permission system is the next line of defense for write-style subcommands — but read-style subcommands that are already permitted in the downstream (e.g., `android --version`) will still fire before failing with "command not found".
- **`android` on PATH but unexecutable** (wrong permissions, broken symlink, corrupt install, wrong-arch binary): `command -v android` returns the path successfully; `NOT_INSTALLED` is never emitted; the skill activates; the first actual invocation fails with an exec error. The guard does not catch this; document it as a known limitation and rely on the exec-error feedback loop.
- **Local `android` CLI version differs from v0.0.2's embedded help dump** (`0.7.15232955`): the guard passes (presence check only); the skill activates; vendored subcommand/flag documentation may not match the user's local CLI behavior. No mitigation in this PR — see Non-Requirements. If the version drift is material, the curator should trigger an out-of-band refresh.
- **`` !`<command>` `` injection unsupported at skill-body scope** (discovered at implementation time): per Technical Constraints, implementer verifies the mechanism works for SKILL.md bodies; if not, fall back to the prose-paragraph alternative defined in Req 6. The fallback does not block this PR — the guard's effective mechanism is the one that survives verification.
- **DAC URL returns HTTP error on a future refresh**: the curator falls back to `github.com/android/skills` release zip at the last-matching tag. HOW-TO-SYNC explicitly documents this fallback in the Upstream source section.
- **Upstream adds a new skill (neither covered nor deferred)**: the `android skills list` reconciliation check-step surfaces the mismatch. The curator must decide whether to cover or defer before completing sync; no silent skip.
- **Upstream bumps `android` CLI version** (e.g., `0.7.15232955` → newer): the vendored `android-cli/SKILL.md` lines 62–196 (literal `android help` output) become version-stale. This ticket does NOT fix that; it only documents the refresh obligation in the `### Accepted divergences` entry. A future re-sync pulls the newer `android help` dump verbatim.
- **CFA-PATCH marker is stripped by a verbatim overwrite on future re-sync**: the central HOW-TO-SYNC `### Accepted divergences` section still documents the guard, allowing the curator to reapply from there. The marker is defense-in-depth, not the authoritative record.
- **Upstream v0.0.2 ships a NOTICE file** (not present as of this research): the curator copies it verbatim into `plugins/android-dev-extras/NOTICE` per Apache 2.0 §4(d). Verify at fetch time during implementation.
- **Plugin validation surfaces unexpected warnings on android-cli**: investigate before committing; only warnings that match existing patterns on `r8-analyzer`/`edge-to-edge` are acceptable without change.

## Technical Constraints

- **Apache 2.0 §4(b)** — modified files must carry prominent notice of modification. Satisfied by (i) the `CFA-PATCH` inline HTML comment in the modified `android-cli/SKILL.md` (Requirement 7) and (ii) the explicit `### Accepted divergences` entry in HOW-TO-SYNC naming the file and change (Requirement 8). **Durability**: (i) can be stripped by a verbatim upstream overwrite on future re-sync — this is exactly the failure mode Requirement 12 prevents (sync procedure's divergence-preservation branch reapplies both marker and guard after update, and the post-update safety check catches cases where reapplication was skipped). Without Req 12, a subsequent sync could silently ship a modified file without its §4(b) notice — a license regression, not merely a feature regression.
- **Apache 2.0 §4(d)** — upstream NOTICE attributions must carry forward. As of the research, upstream `android/skills` v0.0.2 does not ship a NOTICE file, so `plugins/android-dev-extras/NOTICE` (plugin-authored per §4(d)) is unchanged. Re-verify at fetch time in implementation; if upstream v0.0.2 has added a NOTICE, merge verbatim.
- **Plugin validator** — `python3 scripts/validate-skill.py plugins/android-dev-extras/skills` exit 0 is a CI hard gate. Required frontmatter fields: `name`, `description` (both already present in upstream `android-cli/SKILL.md`).
- **Claude Code skill spec** — only documented frontmatter fields may be used. The dynamic context injection mechanism `` !`<command>` `` is documented in [code.claude.com/docs/en/skills](https://code.claude.com/docs/en/skills) for in-body use. Its skill-body semantics were NOT independently verified by the research (observed examples of `` !`<cmd>` `` in the installed plugin set are in `commands/` — slash-command bodies — not in skills). **Implementation must verify** that the `` !`<command>` `` form is interpolated at skill load time in SKILL.md bodies before committing the guard; if verification fails, use the prose-paragraph fallback defined in Req 6 and update HOW-TO-SYNC's `### Accepted divergences` entry to reflect the mechanism actually shipped. In either case, the guard is advisory — Claude Code's loader does not provide a binary-precondition gate; enforcement is via model compliance with the abort instruction. Do NOT add a `preconditions:` frontmatter field to the vendored skill — the loader silently ignores unknown fields, so it provides no enforcement and would be false comfort.
- **Structural invariants** — no new fields in `.claude-plugin/plugin.json`; no new hooks or slash-command declarations; no plugin-level README.md; no modifications to the three sibling plugins.
- **Vendoring source for this PR** — content pulled from `github.com/android/skills` at tag `v0.0.2` (stable git ref for reproducibility). HOW-TO-SYNC documents DAC as the authoritative policy primary; this tactical fetch-from-GitHub is an implementation choice for the initial vendoring.
- **Path-flattening** — upstream `system/android-cli/SKILL.md` → `plugins/android-dev-extras/skills/android-cli/SKILL.md` (category prefix `system/` dropped per existing HOW-TO-SYNC rule).
- **Downstream permission contract** — `chickfila-android` already permits `android --version`, `describe`, `docs`, `list`, `command -v` and `adb -s emulator-5554 shell pm list packages`. New subcommands invoked by the vendored skill (`android screen capture`, `android screen resolve`, `android emulator`, `android run`) are NOT permitted there and will prompt on first use. Not a spec requirement to update downstream allowlists in this PR.

## Open Decisions

- **Skill-body `` !`<command>` `` interpolation semantics** — the research verified the mechanism is documented in the Claude Code skills spec for in-body use, but observed plugin-set examples are from slash-command bodies, not skill bodies. Implementation must run a minimal verification (e.g., add the `!`command -v android`` block to a test copy of the vendored SKILL.md, trigger the skill, and confirm Claude sees the injected output in context) before committing the guard. Cannot be resolved at spec time because verification requires running a skill — an implementation-level check. If verification fails, Req 6's prose-paragraph fallback is used and HOW-TO-SYNC's `### Accepted divergences` entry documents the fallback mechanism instead.
- **Upstream v0.0.2 NOTICE presence** — the research found no NOTICE in the upstream DAC bundle or the `github.com/android/skills` v0.0.2 tag as of 2026-04-22, but this must be re-verified at fetch time during implementation. If upstream has added a NOTICE between research and implementation, Apache 2.0 §4(d) requires copying it verbatim into `plugins/android-dev-extras/NOTICE`. Cannot be resolved at spec time because the authoritative check runs against the fetched content, not against any pre-fetched copy.
